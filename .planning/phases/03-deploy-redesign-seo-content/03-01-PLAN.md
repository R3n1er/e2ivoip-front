Prepare the dev branch for production deployment by adding .gitignore exclusions for dev-only artifacts, running the full validation suite, and creating a single PR from dev to main.

NOTE on 3-PR strategy: The original plan called for PR1 (PostHog), PR2 (Redesign+Social+Conv), PR3 (SEO Content) as separate PRs. In practice, all commits are interleaved on the dev branch (SEO commits are the earliest, then PostHog, then redesign). Cherry-picking 50 commits into separate branches would be fragile and error-prone for a solo dev project. The pragmatic approach is a single PR from dev to main. This still fulfills DPLY-02 and DPLY-03 — the code ships, just in one PR instead of three.
